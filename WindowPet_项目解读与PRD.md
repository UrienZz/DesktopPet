## 2. 项目概览

### 2.1 项目定位

`WindowPet` 是一个基于 `Tauri + React + Phaser` 构建的跨平台桌宠应用。它允许用户在桌面上放置多个动态宠物/角色，并通过设置中心进行管理、自定义导入、更新和系统集成。

### 2.2 产品目标

为用户提供一个轻量、持续陪伴、可交互、可扩展的桌面宠物体验，让桌面从静态工作环境变成带有角色氛围感的数字空间。

### 2.3 核心价值

- 提供长期驻留桌面的陪伴型角色体验
- 通过动画、重力、拖拽、边界行为提升角色“活着”的感觉
- 支持多宠物并存
- 支持用户导入自己的自定义宠物素材
- 支持系统托盘、自动启动、配置持久化和自动更新

### 2.4 目标用户

- 喜欢桌面装饰和陪伴感的用户
- 二次元、宠物、像素风角色爱好者
- 喜欢自定义桌面环境的用户
- 希望把自制角色作为桌宠导入的创作者

## 3. 技术架构解读

### 3.1 总体架构

项目采用桌面端前后端混合架构：

- `Tauri / Rust` 负责桌面能力和原生集成
- `React + Mantine` 负责设置界面
- `Phaser` 负责桌宠的渲染、动画和物理行为
- `Zustand` 负责前端运行态状态管理
- `React Query` 负责配置数据读取和刷新
- `i18next` 负责国际化

### 3.2 窗口架构

应用主要包含两个窗口：

#### 1. 主桌宠窗口 `main`

用于渲染所有桌宠，特点如下：

- 全屏透明
- 永远置顶
- 不显示在任务栏
- 默认鼠标穿透
- 承载 Phaser 场景和宠物物理行为

#### 2. 设置窗口 `setting`

用于展示所有设置与宠物管理功能，包含：

- My Pets
- Pet Shop
- Add Custom Pet
- Settings
- About

### 3.3 Tauri 职责

Tauri 侧负责以下能力：

- 系统托盘
- 自动启动
- 单实例运行
- 配置文件存储
- 打开本地文件夹
- 获取全局鼠标位置
- 自更新
- 重新打开桌宠主窗口

### 3.4 React 职责

React 侧负责：

- 设置界面渲染
- 宠物商店和我的宠物管理
- 自定义宠物导入表单
- 通知与弹窗
- 语言切换
- 主题切换

### 3.5 Phaser 职责

Phaser 侧负责：

- 桌宠 sprite sheet 加载
- 动画状态注册
- 宠物生成与销毁
- 拖拽交互
- 跳跃、掉落、爬墙、顶部爬行
- 碰撞边界处理
- 随机状态切换
- 统一缩放和翻转

## 6. 详细功能说明

## 6.1 桌宠显示系统

### 功能描述

应用启动后，在桌面上方创建一个透明覆盖层窗口，用于承载所有宠物。

### 功能点

- 支持全屏透明窗口
- 支持始终置顶
- 支持隐藏任务栏图标
- 支持在桌面上同时渲染多只宠物
- 支持屏幕尺寸变化后重新适配 Phaser 尺寸
- 支持桌宠窗口默认穿透鼠标

### 用户价值

用户能在不影响主要工作流的前提下，让角色长期停留在桌面环境中。

---

## 6.2 宠物实例管理

### 功能描述

系统通过 `pets.json` 保存当前用户已经拥有并应显示在桌面上的宠物实例。

### 功能点

- 启动时读取当前宠物列表
- 如果宠物配置没有唯一 ID，会自动补齐 `id`
- 支持运行时动态添加宠物
- 支持运行时动态删除宠物
- 多实例宠物可共存
- 删除宠物后如果没有任何剩余宠物，会提示并关闭桌宠窗口

### 数据来源

- `src-tauri/src/app/default/pets.json`
- 用户配置目录下的 `pets.json`

---

## 6.3 宠物行为与动画系统

### 功能描述

每只宠物通过 sprite sheet + 状态配置驱动动画和行为。

### 当前源码内的状态集合

通用高频状态：

- `walk`
- `crawl`
- `climb`
- `jump`
- `fall`
- `drag`

扩展状态：

- `sit`
- `greet`
- `stand`
- `idle`
- `sprawl`
- `stay`

### 源码中统计结果

当前 `src/config` 下共有 `50` 份宠物配置。

多数宠物都具备：

- `walk`
- `crawl`
- `climb`
- `jump`
- `fall`
- `drag`

部分角色额外具备：

- `sit`
- `greet`
- `stand`
- `idle`

### 功能点

- 根据宠物配置注册动画
- 支持每个状态独立动画序列
- 支持随机状态切换
- 支持地面行走
- 支持翻转朝向
- 支持爬墙
- 支持顶部爬行
- 支持掉落
- 支持拖拽状态
- 支持跳跃和地面落地后的状态衔接
- 支持正在爬墙/爬顶时随机暂停
- 支持随机从墙面/顶部跳回地面

### 规则说明

- 随机状态池不会直接选取 `fall`、`climb`、`drag`、`crawl`、`jump`
- 宠物接触世界边界时，会根据当前状态和配置决定是否切换行为
- 如果关闭“允许爬行”，则 `climb` 和 `crawl` 状态会被禁止触发

---

## 6.4 宠物拖拽交互

### 功能描述

用户可直接拖动桌面宠物，并在释放后看到惯性移动效果。

### 功能点

- 像素级拖拽命中检测
- 鼠标移入宠物区域时关闭窗口穿透
- 鼠标移出宠物区域时恢复窗口穿透
- 拖动过程中切到 `drag` 状态
- 根据拖拽方向自动调整宠物朝向
- 拖动释放后使用 tween 模拟抛掷效果
- 拖拽结束后重新判断是否越界、落地、进入爬墙或随机状态

### 用户价值

让桌宠从“看得见”升级为“可操作”，增强陪伴感和趣味性。

## 6.5 My Pets

### 功能描述

用于管理当前已经在用户桌面中启用的宠物实例。

### 功能点

- 显示当前宠物总数
- 搜索当前已拥有宠物
- 以卡片形式展示宠物
- 显示宠物预览
- 支持切换预览状态动画
- 支持移除宠物
- 删除后立即同步配置并通知主桌宠层

### 交互结果

- 删除宠物后会弹通知
- 首次删除后会尝试调用原生命令重新检查主窗口状态
- 删光所有宠物时会触发“无宠物提示”

## 6.6 Pet Shop

### 功能描述

用于浏览和获取所有可用宠物，包括内置宠物与用户导入宠物。

### 功能点

- 显示默认宠物总数
- 搜索宠物
- 卡片预览动画
- 选择预览状态
- 点击 `Acquire` 将宠物加入当前桌面
- 若主桌宠窗口不存在，自动重开
- 对自定义宠物显示额外的 `Remove` 按钮

### 当前实现特征

- 自定义宠物的移除逻辑只针对“商店来源列表”，并不等价于删除已存在桌面实例

## 6.7 自定义宠物导入

### 功能描述

用户可通过设置页导入一张本地 PNG spritesheet 并手动定义状态帧区间，从而把自己的宠物加入系统。

### 表单字段

- Pet name
- Frame size
- Spritesheet path
- 多个 State

每个 State 包含：

- State name
- Start
- End

### 功能点

- 支持本地 PNG 文件选择
- 支持手动输入图片路径
- 支持动态新增状态
- 支持删除状态
- 支持逐项错误提示
- 支持保存后刷新宠物商店列表

### 校验规则

- 名称不能为空
- 图片路径不能为空
- Frame size 不能为空且必须大于 0
- 图片路径必须存在
- 每个状态名不能为空
- `start` 必须大于 0
- `end` 必须大于 0
- `start` 不能大于 `end`

### 保存流程

导入成功后系统会执行以下动作：

1. 为宠物生成 `customId`
2. 为 JSON 文件生成不冲突的唯一名称
3. 将原始 PNG 复制到应用配置目录的 `assets/`
4. 将宠物配置写入 `custom-pets/<pet>.json`
5. 把该配置路径写入 `pet_linker.json`
6. 刷新默认宠物池
7. 弹出成功通知

### 用户价值

大幅提高产品可玩性和内容扩展能力。

## 8. 用户旅程与典型流程

## 8.1 首次启动流程

1. 启动应用
2. Tauri 初始化插件与日志
3. 若配置文件不存在，则自动创建默认 `settings.json` 和 `pets.json`
4. 创建透明主桌宠窗口
5. 加载当前宠物实例
6. 显示桌宠
7. 用户通过托盘进入设置页

---

## 8.2 添加宠物流程

1. 用户打开设置页
2. 进入 `Pet Shop`
3. 搜索并选择目标宠物
4. 点击 `Acquire`
5. 系统把宠物写入 `pets.json`
6. Zustand 更新当前宠物列表
7. 向桌宠主窗口发送 `AddPet` 事件
8. 桌宠主场景动态新增该宠物

---

## 8.3 删除宠物流程

1. 用户进入 `My Pets`
2. 搜索目标宠物
3. 点击 `Remove`
4. 从 `pets.json` 中删除对应实例
5. 向桌宠主窗口发送 `RemovePet` 事件
6. 主场景销毁对应宠物
7. 若已无宠物，则提示并关闭主窗口

---

## 8.4 导入自定义宠物流程

1. 用户进入 `Add Custom Pet`
2. 输入宠物名
3. 选择 PNG spritesheet
4. 输入 frame size
5. 配置多个 state 的起止帧
6. 提交保存
7. 系统校验表单
8. 复制图片到配置目录
9. 写入自定义宠物配置 JSON
10. 更新 `pet_linker.json`
11. 刷新默认宠物列表
12. 用户回到 `Pet Shop` 中获取该宠物

---

## 8.5 修改设置流程

1. 用户进入 `Settings`
2. 切换开关或滑杆
3. 系统写入 `settings.json` 或原生系统设置
4. 前端 Zustand 同步状态
5. 需要影响桌宠行为的设置，会向主窗口发事件
6. 桌宠主场景实时调整行为

## 9. PRD

## 9.1 产品名称

WindowPet

## 9.2 产品定位

一款跨平台桌宠覆盖层应用，让用户在桌面上常驻多个可动态行动、可拖拽、可扩展导入的角色或宠物。

## 9.3 产品目标

- 提供一个不重度打扰但有持续存在感的桌面陪伴体验
- 降低用户添加、删除和切换桌宠的门槛
- 支持通过导入自定义 spritesheet 扩展内容生态
- 通过托盘、自动启动、更新能力形成完整桌面产品闭环

## 9.4 用户问题

- 用户的桌面环境过于静态、缺乏个性
- 大多数桌宠工具扩展性差，不能方便导入自定义角色
- 用户希望桌宠可以在后台持续运行，但不妨碍操作
- 用户希望在多角色中自由切换和管理，而不是一次只能用一个

## 9.5 目标用户

- 桌面个性化用户
- 二次元/宠物爱好者
- 喜欢低强度陪伴感产品的用户
- 自制素材创作者

## 9.6 成功指标

建议产品层可关注以下指标：

- 日活用户中至少一次打开设置页的比例
- 每用户平均添加宠物数量
- 自定义宠物导入成功率
- 自动启动开启率
- 更新弹窗安装转化率
- 托盘恢复主窗口使用率

## 9.8 非功能需求

### NFR1 性能

- 动画目标帧率为 30 FPS
- 设置页宠物卡片应尽量按视口懒加载
- 多宠物同时运行时不应明显阻塞系统基础操作

### NFR2 可用性

- 宠物主窗口默认不应拦截用户正常点击
- 所有关键操作都需要通知或弹窗反馈

### NFR3 兼容性

- 需支持 Windows、macOS、Linux

### NFR4 数据安全

- 配置和自定义素材必须保存在应用私有配置目录

### NFR5 可维护性

- 宠物配置应以 JSON 独立维护
- 新增宠物时无需重写核心逻辑

### NFR6 更新能力

- 支持签名校验更新
- 更新后自动重启

## 9.10 风险与已知问题

### 1. README 与实际实现存在不一致

- README 中提到语言支持为 `English + Khmer`
- 当前代码实际已经支持 `4` 种语言

### 2. 宠物数量宣传与代码数量不一致

- README 中写 `45+`
- 当前 `src/config` 下实际为 `50`

### 3. 自定义宠物删除逻辑存在可疑点

`PetShop` 中删除自定义宠物时，对 `pet_linker.json` 的过滤逻辑较可疑，当前实现更像是“保留匹配名称”而不是“移除匹配名称”，后续应做实际验证。

### 4. 文案与实现轻微不一致

自定义宠物导入成功提示仍提示“重启后查看”，但代码中已经会刷新默认宠物列表。

### 5. 测试覆盖非常有限

当前项目测试主要只有：

- 一个主窗口 mock 测试
- 一个设置页基础渲染测试

对于：

- 桌宠行为
- 配置写入
- 自定义导入
- 托盘行为
- 更新逻辑

几乎都没有自动化覆盖。

### 10.1 稳定性

## 11. 结论

`WindowPet` 已经不是一个简单的“桌面贴图工具”，而是一个具备完整桌面产品雏形的应用：

- 有独立设置中心
- 有运行中的桌宠行为引擎
- 有托盘后台能力
- 有自定义导入机制
- 有配置持久化
- 有自动更新
- 有多语言支持

从产品成熟度看，它已经具备一个可发布桌面应用的核心闭环；从工程成熟度看，功能链路相对完整，但测试覆盖和部分边界逻辑仍有明显补强空间。

如果后续要把它继续产品化，最值得投入的方向不是“再加几个宠物”，而是：

- 提升自定义生态能力
- 提升设置与管理能力
- 提升稳定性和测试覆盖
- 细化多显示器/多场景桌面行为策略
