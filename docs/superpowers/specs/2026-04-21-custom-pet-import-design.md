# 自定义导入宠物设计

## 背景

当前应用只支持加载内置宠物资源。用户可以在设置页切换已有宠物，但无法导入自定义宠物，也无法删除用户后续添加的宠物资源。

本次扩展目标是支持导入一个包含宠物 `json` 配置和 `png` 精灵图的压缩包，在解析通过后把该宠物加入设置页与桌面运行时；同时允许删除已导入的宠物，并在删除当前正在使用的导入宠物时自动回退到内置默认宠物。

## 目标

- 支持通过设置页导入一个 `.zip` 宠物包。
- 宠物包合法时完成解压、校验、持久化，并加入宠物列表。
- 宠物包不合法时给出明确错误提示，且不产生脏数据。
- 支持删除导入宠物。
- 删除当前正在使用的导入宠物时，立即切换到默认内置宠物并刷新桌面显示。
- 重启应用后仍能恢复已导入宠物。

## 非目标

- 不支持在线下载宠物。
- 不支持一个 zip 导入多个宠物。
- 不支持导入时重命名宠物。
- 不支持编辑宠物配置。
- 不支持删除内置宠物。
- 不支持压缩包内的图片格式扩展到 `webp/gif`，本次仅要求 `png`。

## 用户体验

### 导入流程

1. 用户进入“设置 > 桌宠”。
2. 点击“导入宠物”按钮。
3. 弹出 macOS 原生文件选择器，只允许选择 `.zip`。
4. 应用解压并校验 zip 内容。
5. 成功时：
   - 弹出成功提示，内容为“已导入宠物：xxx”。
   - 宠物列表立即刷新。
   - 新宠物可立即被选中使用。
6. 失败时：
   - 弹出失败提示，内容为“宠物包不合法，请检查 json 配置和 png 资源”。
   - 不写入任何导入结果。

### 删除流程

1. 用户在“设置 > 桌宠”中选择一只导入宠物。
2. 页面显示“删除该宠物”危险按钮。
3. 用户点击后删除该导入宠物目录与注册信息。
4. 如果当前正在使用这只导入宠物：
   - 立即切换到默认内置宠物。
   - 桌面桌宠立刻刷新为默认宠物。
   - 偏好存储中的当前宠物名称同步更新。

## 数据模型

### 运行时宠物条目

当前 `PetDefinition` 只承载宠物配置本身，不区分来源。为了支持删除与来源标识，需要在运行时引入一层包装模型，例如：

- `PetCatalogEntry`
  - `definition: PetDefinition`
  - `source: .bundled | .imported`
  - `storageURL: URL?`
  - `isDeletable: Bool`

运行时界面和协调器统一面向 `PetCatalogEntry` 工作，底层桌宠渲染仍使用 `PetDefinition`。

### 导入目录结构

应用在自己的支持目录下维护导入宠物目录，例如：

`Application Support/DesktopPetApp/ImportedPets/<uuid-or-slug>/`

每个导入宠物目录至少包含：

- `pet.json`
- 精灵图 `*.png`

允许 zip 内文件名保持原样，但导入后会统一重命名或标准化到应用自己的目录结构，避免依赖外部压缩包路径。

## 架构设计

### 1. 宠物导入存储层

新增 `ImportedPetStore`，负责：

- 选择 zip 后的解压工作目录管理
- 导入目录创建
- 配置与资源校验
- 持久化导入宠物
- 删除导入宠物
- 列出所有已导入宠物

该层不关心设置页和桌面状态，只负责文件系统与校验。

### 2. 宠物目录加载层

扩展 `PetCatalogLoader`：

- 保留现有内置宠物加载
- 新增导入宠物加载
- 输出统一的 `PetCatalogEntry` 列表

排序规则：

- 先内置、后导入，或统一按名称排序

推荐：对 UI 保持简单，统一按名称排序；来源信息通过附加标签展示。

### 3. 协调器层

扩展 `AppCoordinator`：

- 持有 `[PetCatalogEntry]`
- 暴露当前宠物是否可删除
- 提供：
  - `importPetArchive()`
  - `deleteSelectedImportedPet()`
  - `reloadPetCatalog()`
- 导入成功后刷新宠物列表
- 删除当前使用的导入宠物时回退默认内置宠物

### 4. 设置页层

扩展“设置 > 桌宠”：

- 顶部摘要区增加“导入宠物”按钮
- 当前宠物卡中显示来源标识：
  - “内置”
  - “自定义”
- 当选中的是导入宠物时，显示“删除该宠物”危险按钮
- 删除按钮仅对导入宠物显示

## 导入校验规则

zip 解析通过的最低条件：

- 压缩包中存在且仅处理一个宠物 `json`
- `json` 能解码为合法 `PetDefinition`
- `states` 不为空
- `imageSrc` 指向的 `png` 文件存在
- 对应 `png` 文件可从解压目录正确定位

校验失败统一视为非法宠物包，不做部分导入。

## 文件处理流程

1. 用户选中 zip。
2. 将 zip 解压到临时目录。
3. 扫描临时目录，查找 `json`。
4. 解析 `PetDefinition`。
5. 根据 `imageSrc` 定位对应 `png`。
6. 通过全部校验后，把 `json + png` 复制到应用导入目录。
7. 清理临时目录。

## 删除策略

删除时按 `PetCatalogEntry.storageURL` 删除其对应目录。

如果删除的是当前宠物：

1. 找到默认内置宠物。
2. 调用现有桌宠切换逻辑切回默认宠物。
3. 更新偏好存储。
4. 刷新窗口渲染。
5. 再移除导入目录与列表项。

这样可以避免先删文件导致当前渲染资源失效。

## 错误处理

### 用户可见错误

- 文件类型错误：选择的不是 zip
- 解压失败：压缩包损坏或系统无法展开
- 配置非法：json 无法解码
- 资源缺失：找不到 json 指定的 png
- 删除失败：目录不可访问或文件锁定

对用户统一提示简洁中文信息，详细错误只用于调试日志。

### 内部保护

- 导入采用临时目录，失败不污染正式导入目录
- 正式写入前先做完整校验
- 删除时只允许删除导入目录，不允许碰内置资源目录

## UI 变更

### 桌宠页摘要区

- 新增 `导入宠物`
- 当当前宠物为导入宠物时显示 `删除该宠物`

### 当前宠物卡

- 显示当前宠物名称
- 显示来源标签

### 宠物选择器

- 列表中导入宠物可附加“自定义”标识

## 测试计划

### 单元测试

- 合法 zip 可成功导入
- 非法 zip 不会写入导入目录
- 缺少 png 时导入失败
- 删除导入宠物后会从列表移除
- 删除当前使用的导入宠物时会回退默认内置宠物
- 重载 catalog 时同时包含内置与导入宠物

### 集成回归

- 导入后可在设置页选中并切换到桌面桌宠
- 重启应用后导入宠物仍存在
- 删除后设置页与桌面状态一致

## 实现拆分建议

1. 新增导入存储层和导入目录约定
2. 扩展 catalog loader 支持导入宠物
3. 扩展 coordinator 处理导入、删除、回退默认宠物
4. 更新“桌宠”设置页按钮与来源展示
5. 补充测试与错误提示

## 风险

- zip 解压能力需要选择稳定实现方式；若系统原生归档接口受限，需要确认当前项目依赖策略
- 当前 `PetDefinition.mediaFileURL` 依赖全局资源目录，导入宠物后需要改造成基于来源目录解析资源路径，否则内置与导入资源会混淆
- 若用户导入同名宠物，UI 展示层可能出现重名，需要在来源标签或内部唯一目录策略上规避混淆

## 结论

推荐采用“导入后托管到应用支持目录”的方案。它最适合当前桌宠的持久化、删除、重启恢复和资源安全控制，也能最小化后续在设置页和运行时行为上的不确定性。
